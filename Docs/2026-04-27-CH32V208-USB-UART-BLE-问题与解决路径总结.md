# CH32V208 USB UART BLE 项目问题与解决路径总结

## 1. 文档目的

本文档汇总本项目推进到当前阶段已经遇到的主要问题、定位证据、解决路径和当前状态。

当前项目目标是基于 `CH32V208` 实现一个单 `USB Vendor Interface` 设备，通过私有协议同时承载：

- `4` 路虚拟串口能力
- `BLE Host` 扫描、连接、GATT 访问能力
- 系统管理、心跳、日志和后续扩展通道

本文档只记录已经在当前仓库或实板路径中出现过的问题，不替代设计文档和实施计划。

## 2. 当前阶段状态

### 2.1 固件侧

- USB 设备可以稳定枚举为 `1a86:2080`
- USB 描述符为单一 `Vendor Specific Interface`
- 当前端点约定为：
  - `EP3 OUT 0x03`：Host 到 Device 帧
  - `EP2 IN 0x82`：Device 到 Host 帧
  - `EP1 IN 0x81`：异步 hint
- 协议头固定为 `24` 字节，`magic = 0xA55A`，`version = 0x01`
- 固件已具备 `SYS`、`UART_CTRL`、`UART_DATA`、`BLE_MGMT`、`BLE_CONN` 通道
- 虚拟串口 `0` 已完成物理回环验证
- BLE 扫描已经可以通过 Host CLI 调用并接收广播事件

### 2.2 Host 侧

`Host/` 子项目是独立 C/CMake 工程，使用 `libusb-1.0` 直接对接当前固件。

当前 CLI 已覆盖：

- `probe`
- `heartbeat`
- `debug-xfer`
- `uart-cap`
- `uart-open`
- `uart-close`
- `uart-loopback`
- `ble-scan`

已验证命令包括：

```sh
cmake --build build
ctest --test-dir build --output-on-failure
./build/ch32v208-mux-cli probe
./build/ch32v208-mux-cli uart-loopback 0 115200 "payload"
./build/ch32v208-mux-cli ble-scan 3
```

## 3. USB 枚举阶段问题

### 3.1 Host 完全看不到设备

**现象**

Host 侧 `lsusb` 看不到目标设备，内核日志出现类似：

```text
Cannot enable
unable to enumerate USB device
```

**定位判断**

此阶段还没有进入 Host 驱动或协议层，优先判断为设备侧 USB bring-up 问题。

关键点是 CH32V208 USBFS 需要同时完成：

- USB 时钟配置
- USBFS 控制器初始化
- USB 端口使能
- D+ 上拉使能

**解决路径**

- 对照 WCH `BLE_USB` 参考例程检查初始化顺序
- 确认 `96MHz SYSCLK` 下 USB 分频使用 `RCC_USBCLKSource_PLLCLK_Div2`
- 增加 `EXTEN->EXTEN_CTR |= EXTEN_USBD_PU_EN`
- 在 USB 初始化后延时并再使能控制器和中断

**当前状态**

已解决。设备可以枚举为：

```text
1a86:2080 QinHeng Electronics CH32V208 Mux Dev
```

### 3.2 Host 能发现 full-speed 设备但读描述符超时

**现象**

Host 侧能看到新 full-speed USB 设备，但出现：

```text
device descriptor read/64, error -110
device descriptor read/64, error -71
```

**定位判断**

这说明 USB 物理连接和上拉已经生效，EP0 控制传输已经到达 MCU，但 EP0 SETUP/IN/OUT 处理存在问题。

曾经的关键线索是 `INT_ST` 中出现 SETUP token 活动，但低位端点值不一定等于 `0`。如果代码只在 `endpoint == 0` 时处理 SETUP，就可能漏掉实际 SETUP 分支。

**解决路径**

- SETUP 处理改为优先按 token 判断
- EP0 IN/OUT 再按端点和 token 单独处理
- 修正描述符、地址设置、配置设置的 EP0 状态机
- 保持 EP0 控制路径的 ACK/NAK/STALL 行为明确

**当前状态**

已解决。`lsusb -v -d 1a86:2080` 可以正常读取设备、配置、接口和端点描述符。

## 4. USB 数据端点与 Host 协议问题

### 4.1 Host `probe` 超时

**现象**

设备已经枚举，Host 可以 claim interface，但 `probe` 或 `GET_DEV_INFO` 超时。

低层 `debug-xfer` 显示：

```text
bulk OUT ep=0x03 ret=OK transferred=24
bulk IN ep=0x82 ret=ERR_TIMEOUT transferred=0
```

**定位判断**

这说明 Host 到 Device 的 OUT 端点已经能写入，但 Device 到 Host 的响应没有正常从 IN 端点返回。

实际排查时需要区分三类原因：

- Host 和固件端点方向约定不一致
- 固件已经排队响应，但 IN 端点没有正确发送
- 固件镜像不是当前协议实现，设备只是枚举成功但没有运行当前业务路径

**解决路径**

- 固定 Host/固件端点约定为 `EP3 OUT` 和 `EP2 IN`
- 使用 `debug-xfer` 验证单次原始 bulk OUT/IN
- 使用串口日志确认固件是否进入 `USBRX -> VendorRouter -> USBTX`
- 发现板上固件状态不一致时，重新 `make flash` 烧录当前镜像

**当前状态**

已解决。重新烧录当前固件后：

```text
proto=1 fw=0.1.0 uart_ports=4 ble_links=3
```

### 4.2 首次传输成功、后续传输超时

**现象**

Host 第一笔事务可能成功，但连续执行 `probe`、`heartbeat` 或其他命令时后续超时。

**定位判断**

此类问题通常不是描述符或枚举问题，而是端点重装载后的 DATA toggle 或 ACK/NAK 状态没有保持正确。

**解决路径**

- EP2 IN 完成后重回 NAK 时保留 `USBFS_UEP_T_TOG`
- EP3 OUT 重新 ACK 时保留 `USBFS_UEP_R_TOG`
- 拆分 EP2 IN 和 EP3 OUT DMA buffer，避免共享缓冲引入状态污染
- 用多次 `debug-xfer`、`probe`、`heartbeat` 验证重复事务

**当前状态**

已解决。Host 工具可以连续执行 `probe`、UART 和 BLE 命令。

## 5. 日志与运行时调度问题

### 5.1 早期缺少应用层日志

**现象**

USB、BLE、UART 应用层行为不可观察，只能靠 Host 侧超时推测。

**解决路径**

- 增加应用层 `APP_LOG`
- 增加启动 banner，输出 Git 描述、构建日期和时间
- 将 USB、BLE、UART 关键状态路径打点
- 默认调试输出走 `SDI_PRINT`

**当前状态**

已解决。固件启动和关键路径具备基本可观测性。

### 5.2 日志直接打印影响热路径

**现象**

USB IRQ、RX、TX 等高频路径大量打印，可能造成时序噪声和 TMOS 调度压力。

**解决路径**

- 日志改为 RAM ring buffer
- 增加独立 TMOS log task
- flush 时直接从 ring buffer 区域写出，避免额外 pop 到临时 buffer
- 增加日志等级：
  - `DEBUG`
  - `INFO`
  - `WARNING`
  - `ERROR`
- 默认最小等级为 `INFO`
- USB low-level 高频日志降为 `DEBUG`
- 对最热 USB IRQ/RX/TX 日志增加编译期开关

**当前状态**

已解决。`make test-log-levels` 覆盖日志等级和部分宏约束。

## 6. Heartbeat 长时间运行问题

### 6.1 运行数分钟后心跳停止

**现象**

心跳运行一段时间后停止。GDB 单步仍能看到系统在 `TMOS_SystemProcess()` 及相关 ROM helper 内部运行。

**定位判断**

不能直接把问题归咎于 TMOS 内部函数。实际有两个方向：

- 心跳时间基准计算存在 32 位溢出
- 高频 USB/log 任务可能放大 TMOS 单回调调度下的饥饿问题

已确认的具体 bug 是：

```text
rtc_ticks * 1000U
```

在 32 位中间值中约数分钟后会溢出。

**解决路径**

- 避免把 RTC tick 先转换为毫秒再调度
- 使用 `next_deadline_tick`，保持调度在 RTC tick 域内
- 增加 `make test-heartbeat` 覆盖溢出场景
- 降低 USB 高频日志对 TMOS 调度的干扰

**当前状态**

已解决。心跳溢出路径已有测试覆盖。

## 7. 临界区与中断控制问题

### 7.1 临界区相关崩溃风险

**现象**

在日志 ring buffer 和共享状态保护中需要禁用/恢复中断。若混用通用 RISC-V `mstatus` 思路和 WCH 当前接口，可能引入运行时崩溃或恢复状态不一致。

**定位判断**

本项目使用的中断使能状态与 `__disable_irq()` / `__enable_irq()` 所操作的 CSR 域必须保持一致。

**解决路径**

- 中断状态保存/恢复使用与 WCH disable/enable primitive 一致的 CSR 域
- 不用泛化的 RISC-V 模板直接替代当前库语义
- 临界区只覆盖最小共享数据访问

**当前状态**

已处理。日志缓冲和任务通信路径使用当前仓库验证过的临界区实现。

## 8. 虚拟串口问题

### 8.1 虚拟串口 0 对应引脚不清晰

**现象**

需要确认 Host 侧虚拟串口编号和实际 MCU UART 引脚关系。

**解决路径**

以 `App/usb_mux_dev/config/uart_map_config.c` 为准：

```text
虚拟串口0 -> USART1 TX PA9   RX PA10
虚拟串口1 -> USART2 TX PA2   RX PA3
虚拟串口2 -> USART3 TX PB10  RX PB11
虚拟串口3 -> UART4  TX PC10  RX PC11
```

**当前状态**

已确认。

### 8.2 虚拟串口回环首次误判失败

**现象**

物理回环 PA9/PA10 后，Host 侧初版 `uart-loopback` 只收到前两个字节：

```text
tx=12 rx=2
RX: 6C 6F
loopback mismatch
```

**定位判断**

固件侧 UART RX 会按当前调度拆成多个 `UART_DATA` 帧返回。Host 初版测试命令只读了一帧，误把分片当成数据丢失。

**解决路径**

- Host CLI `uart-loopback` 改为循环读取
- 累计到期望 payload 长度后再比较
- 再关闭 UART

**当前状态**

已解决。实测：

```text
uart 0 loopback baud=115200 tx=12 rx=12
loopback ok
```

## 9. BLE 扫描问题

### 9.1 固件已有 BLE scan，但 Host 无入口

**现象**

固件侧已经实现：

- `VP_BLE_SCAN_START`
- `VP_BLE_SCAN_STOP`
- `VP_BLE_EVT_SCAN_RSP`

但 Host CLI 只能执行 `probe`、`heartbeat` 和 UART 命令。

**解决路径**

- 在 Host 子项目新增 `ble.h` / `ble.c`
- 增加 BLE 能力查询、扫描参数设置、开始扫描、停止扫描、事件读取和扫描事件解析 API
- CLI 增加：

```sh
./build/ch32v208-mux-cli ble-scan 3
```

- 为扫描事件 payload 解析增加单元测试

**当前状态**

已解决。实测 `ble-scan 3` 可收到大量广播事件。

### 9.2 BLE scan stop 返回设备状态错误

**现象**

初版 Host 命令同时设置固件扫描 duration 和 Host 侧等待时间。扫描可能已经由固件自然结束，Host 再发 `SCAN_STOP` 时返回设备状态错误。

**定位判断**

这是 Host 侧控制策略问题，不是扫描失败。扫描事件本身已经正常返回。

**解决路径**

- 固件扫描 duration 设为 `0`
- 由 Host CLI 自己计时
- 时间到后显式发送 `SCAN_STOP`

**当前状态**

已解决。`ble-scan 3` 可以正常退出，曾实测收到 `154` 条扫描记录。

### 9.3 BLE 输出中的 `type` 和 `event` 含义不直观

**现象**

CLI 当前输出类似：

```text
scan[0] addr=08:C5:D5:35:2A:3E type=2 event=3 rssi=-84 adv_len=31 adv_prefix=1EFF06000109
```

用户需要理解 `type` 和 `event` 字段。

**定位判断**

当前 CLI 原样打印固件从 WCH GAP 扫描事件中转出的：

```text
type  = addrType
event = eventType
```

常见含义：

```text
type=0  Public Device Address / Public Identity Address
type=1  Random Device Address / Random Static Identity Address
type=2  Public Identity Address
type=3  Random Identity Address

event=0  ADV_IND，可连接非定向广播
event=1  ADV_DIRECT_IND，可连接定向广播
event=2  ADV_SCAN_IND，可扫描非定向广播
event=3  ADV_NONCONN_IND，不可连接非定向广播
event=4  低占空比定向广播或扫描响应/补充报告，需结合 WCH 报告类型判断
```

**解决路径**

当前只做原始字段输出。后续可把 CLI 输出增强为可读文本，例如：

```text
addr_type=random_identity adv_type=scan_rsp rssi=-80 name=...
```

**当前状态**

基础扫描已完成；友好解码和广播 AD structure 解析尚未完成。

## 10. Host 子项目和串口日志协作问题

### 10.1 Host 驱动边界容易漂移

**现象**

容易按通用 CDC ACM 或内核 tty 驱动思路设计 Host 侧，而当前固件实际是单 Vendor Interface 私有协议。

**解决路径**

- Host 首阶段采用用户态 `libusb`
- 协议层、传输层、业务 API 分层：
  - `proto.c`
  - `device.c`
  - `uart.c`
  - `ble.c`
- CLI 只调用公开 API
- 后续迁移内核态时复用协议语义，替换传输层

**当前状态**

已建立 `Host/` 独立工程并完成基础 SYS、UART、BLE scan 验证。

### 10.2 串口日志和 minicom 抢占同一个串口

**现象**

用户用 `minicom` 查看日志时，调试脚本再直接打开真实串口会失败或互相抢占。

**解决路径**

- 使用串口 tap 模式
- tap 进程独占真实串口
- 人通过 PTY 继续使用 `minicom`
- 脚本通过 tap 产物读取日志或发送触发

**当前状态**

已形成固定调试模式。后续抓日志时优先沿用 tap/PTY，而不是多个进程争用真实串口。

## 11. 烧录与工具识别问题

### 11.1 OpenOCD 识别容量不符合项目约定

**现象**

OpenOCD 可能将目标识别为：

```text
flash size = 160kbytes
ROM 128 kbytes RAM 64 kbytes
```

并把 flash bank 限制在 `0x08028000` 以下。

**定位判断**

这不应直接推翻项目对 `CH32V208` 的存储映射约定。本项目按 `448KB` 可写 Flash 理解，其中 `128KB` 为快速 Flash 区。

**解决路径**

- 默认烧录使用 `make flash`
- `make flash` 调用仓库内 `Scripts/wch_flash.py`
- 通过 `Scripts/WCH/CommunicationLib/libmcuupdate.so` 烧录 `Out/*.hex`
- 默认地址使用 `0x08000000`
- `make flash-openocd` 仅保留为诊断入口

**当前状态**

`make flash` 已可成功烧录当前镜像。OpenOCD 容量识别问题不作为当前主烧录路径。

## 13. 后续定位原则

### 13.1 先分层，再修复

遇到问题时优先确认卡在哪一层：

```text
USB 枚举
EP0 控制传输
Bulk OUT
固件协议解析
业务路由
TX 队列
Bulk IN
Host 协议解析
业务 API
```

### 13.2 不把 Host 问题和设备问题混在一起

- `lsusb` 不出现：优先看设备侧 pull-up、时钟、USB init
- 描述符读取失败：优先看 EP0
- OUT 成功但 IN 超时：看端点、toggle、固件响应队列
- 响应状态错误：看业务 opcode、状态机和参数
- 数据不完整：先确认是否被分片，不直接判定丢包

### 13.3 实测命令优先

当前最有用的验证命令：

```sh
make
make flash
make test-heartbeat
make test-log-levels

cd Host
cmake -S . -B build
cmake --build build
ctest --test-dir build --output-on-failure
./build/ch32v208-mux-cli probe
./build/ch32v208-mux-cli debug-xfer
./build/ch32v208-mux-cli heartbeat ping
./build/ch32v208-mux-cli uart-loopback 0 115200 "loop0"
./build/ch32v208-mux-cli ble-scan 3
```
