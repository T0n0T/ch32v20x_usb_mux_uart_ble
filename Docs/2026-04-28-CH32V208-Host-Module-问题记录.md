# CH32V208 Host 侧内核模块问题记录

## 1. 文档目的

本文记录当前 `Host/module` 首版 Linux Host 侧内核模块推进过程中暴露的问题、对应处理方式和仍需验证的边界。

记录依据包括：

- 已有用户态 Host 驱动路径：`Host/userspace`
- 当前新增内核模块路径：`Host/module`
- 既有协议与实施文档：`Docs/DESIGN/CH32V208_USB_VENDOR_UART_BLE_HOST_FIRMWARE_DESIGN.md`、`Docs/PLAN/2026-04-24-host-libusb-userspace-driver-plan.md`
- 固件侧当前私有协议头和端点约定：`App/usb_mux_dev/proto/vendor_proto.h`

本文只记录本项目当前路径中已经出现或由当前实现直接暴露的问题，不替代 `Host/module/README.md`。

## 2. 当前变更概况

当前 `Host/module` 是在已经验证过 `Host/userspace` libusb 工具之后推进的内核侧实现，目标是让 Linux 侧出现：

- 字符设备：`/dev/ch32v208-muxN`
- 虚拟串口：`/dev/ttyCH32MUX0` 到 `/dev/ttyCH32MUX3`
- ioctl 事务入口：`CH32MUX_IOC_TRANSACT`
- 异步事件读取入口：`CH32MUX_IOC_READ_EVENT`
- 用户态 BLE 调试工具：`Host/module/tools/ch32v208-ble`

当前模块实现文件主要包括：

- `Host/module/ch32v208-mux-core.c`
- `Host/module/ch32v208-mux-proto.c`
- `Host/module/ch32v208-mux-private.h`
- `Host/module/include/uapi/ch32v208_mux_ioctl.h`
- `Host/module/tools/ch32v208-ble.c`
- `Host/module/tests/test_ioctl_abi.c`
- `Host/module/udev/*.rules`

当前无硬件验证已覆盖：

```sh
make -C Host/module test
make -C Host/module
```

上述两项在当前主机上可以通过，说明 ioctl ABI 静态检查和本机内核头文件下的模块构建路径可用。

## 3. 已遇到的问题与处理

### 3.1 不能直接把用户态 libusb 代码搬进内核模块

**现象**

用户态 `Host/userspace` 已经具备 `probe`、`uart-loopback`、`ble-scan` 等验证命令，但进入内核模块后，传输层不能直接复用。

**原因**

用户态路径依赖 `libusb_init()`、`libusb_claim_interface()`、`libusb_bulk_transfer()`；内核模块路径需要使用 USB core 的 `probe/remove`、`usb_bulk_msg()` 或 URB、内核内存分配和 tty 子系统。

**处理**

- 协议编解码逻辑独立到 `ch32v208-mux-proto.c`
- USB transport 在 `ch32v208-mux-core.c` 中重写
- 用户态工具只保留 BLE 命令编排和 ioctl 调用，不再直接 claim USB interface

**当前状态**

协议层可复用，USB 传输层已经按内核 API 重写。后续若做高频数据路径，仍应从同步 `usb_bulk_msg()` 迁移到 URB + 队列。

### 3.2 内核模块与用户态 libusb 工具会争用同一个 USB interface

**现象**

一旦内核模块绑定 `1a86:2080` 设备，旧的 `Host/userspace` libusb 工具就不应继续同时 claim 同一个 interface。

**原因**

当前设备是单一 USB Vendor Interface，不是 CDC ACM 复合设备。内核模块绑定后，接口所有权已经在 kernel driver 侧。

**处理**

- 验证用户态路径时使用 `Host/userspace`
- 验证内核模块路径时使用 `Host/module/tools/ch32v208-ble` 和 `/dev/ttyCH32MUX*`
- 两条路径不要并发跑；必要时卸载模块后再回到 libusb 工具

**当前状态**

`Host/module/README.md` 已记录这一约束。后续实板测试时，如果出现 libusb `claim` 失败或 ioctl 超时，应先确认接口是否已被另一条路径占用。

### 3.3 私有协议不能按通用 USB 串口或 BLE HCI 处理

**现象**

Host 侧实现容易误以为目标是普通 USB tty、CDC ACM 或 BLE HCI 设备。

**原因**

本项目固件侧枚举为单一 `Vendor Specific Interface`，通过 24 字节私有协议头承载 `SYS`、`UART_CTRL`、`UART_DATA`、`BLE_MGMT`、`BLE_CONN` 等通道。Host 侧必须按私有帧协议解码和分发。

**处理**

- 模块固定使用当前固件协议：`magic = 0xA55A`、`version = 0x01`、`header_len = 24`
- 固定端点方向：`EP3 OUT 0x03`、`EP2 IN 0x82`
- 通过 `seq/ref_seq/ch_type/ch_id/opcode` 匹配同步响应
- BLE 连接、扫描、GATT 命令继续放在用户态工具中编排

**当前状态**

模块已经实现通用帧事务和事件读取；没有在内核中硬编码完整 BLE 业务流程。

### 3.4 EP2 IN 同时承载响应、事件和 UART 数据，必须统一收包再分发

**现象**

用户态早期同步命令模型默认一次命令后等一个响应。但内核模块要同时面对：

- `RSP`：同步 ioctl 命令响应
- `EVT`：BLE 扫描、连接、GATT 事件
- `DATA`：UART 数据流

**原因**

当前固件所有 Device 到 Host 的私有协议帧都走 `EP2 IN`，不能让多个调用点各自阻塞读取同一个 bulk IN 端点。

**处理**

- 模块内增加单一 RX 线程统一读取 `EP2 IN`
- 匹配当前等待的 `RSP` 时唤醒 ioctl 等待者
- BLE `EVT/DATA` 进入 `/dev/ch32v208-mux0` 事件队列
- `UART_DATA` 写入对应 `ttyCH32MUX*` 的 tty flip buffer

**当前状态**

当前事件队列上限为 `32`，满时丢弃最旧事件。该策略适合调试验证，但不适合高频 BLE notify 或高吞吐 UART 场景。

### 3.5 TTY 语义与固件 UART 控制命令之间存在映射缺口

**现象**

Linux tty 层天然期望串口具备 open/close、termios、写缓冲、流控、modem control、break、flush 等行为；当前固件协议只提供项目自定义 UART 控制和数据通道。

**处理**

当前模块先实现最小可用映射：

- 打开 `/dev/ttyCH32MUX*` 时发送 `UART_OPEN`
- 关闭最后一个文件描述符时发送 `UART_CLOSE`
- `termios` 变化映射为 `UART_SET_LINE_CODING`
- tty write 直接封装为 `UART_DATA` 帧并同步发送
- 固件上行 `UART_DATA` 进入 tty flip buffer

**当前边界**

以下能力尚未实现：

- TTY 写侧队列和限流
- modem control
- break
- UART flush
- 高吞吐场景下的 URB 化收发和 backpressure

### 3.6 ioctl ABI 必须固定，否则用户态工具和模块容易漂移

**现象**

内核模块和 `Host/module/tools/ch32v208-ble` 之间需要共享 UAPI 结构体。如果结构体字段、大小或 ioctl 编号漂移，用户态工具会以错误布局传参。

**处理**

- 公共 UAPI 头文件放在 `Host/module/include/uapi/ch32v208_mux_ioctl.h`
- 定义 `CH32MUX_IOC_ABI_VERSION = 1`
- 增加 `Host/module/tests/test_ioctl_abi.c`，检查关键常量、字段偏移和结构体大小

**当前状态**

当前验证命令：

```sh
make -C Host/module test
```

已通过。后续改 UAPI 时，应同步修改 ABI 版本和测试预期。

### 3.7 设备节点权限需要单独处理

**现象**

内核模块注册的 `/dev/ch32v208-muxN` 和 `/dev/ttyCH32MUX*` 不是系统默认串口节点，普通用户可能遇到 `Permission denied`。

**处理**

当前提供两条 udev 规则：

- `Host/module/udev/60-ch32v208-mux-kernel.rules`
- `Host/module/udev/61-ch32v208-mux-tty.rules`

规则把字符设备和 tty 节点设置为 `0660`，组为 `uucp`，并加 `uaccess`。

**当前边界**

udev 规则需要安装到系统目录后才生效；如果设备节点已经存在，通常需要 reload rules 后重新插拔设备或重新加载模块。

### 3.8 当前内核模块仍是功能验证版，不是高吞吐稳定版

**现象**

当前代码可以构建出 `ch32v208_mux.ko`，但实现仍偏向证明协议和设备节点路径可行。

**原因**

首版模块使用同步 `usb_bulk_msg()` 进行 bulk 收发，并通过 RX 线程分发帧。这比直接多处阻塞读端点更清晰，但不是长期高吞吐设计。

**当前边界**

- interrupt hint endpoint 尚未接入
- bulk IN/OUT 尚未 URB 化
- UART 写侧没有排队和限流
- BLE 高频事件队列只有固定长度
- 已完成 BLE scan 和 tty 回环验证，但还不是高频 UART 或 BLE notify 压测结论

## 4. 当前验证状态

已验证：

- `make -C Host/module test`

## 5. 后续建议

1. 将本次实板命令、输出摘要和内核日志补进验证记录，便于后续回归对比。
2. 补充 BLE `cap`、长时间 scan、多设备扫描和 notify 事件验证。
3. 补充多路 tty 并发收发、长包、持续吞吐和异常断开恢复验证。
4. 若基本链路持续稳定，再补充 interrupt hint 和 URB 化收发。
5. 若要长期暴露给普通应用，优先补齐 tty 写侧队列、限流、flush 和异常断开恢复。
6. 提交时注意当前父仓库和 `Host/` 子仓库是分开的，`Host/module` 文件应先在 Host 子仓库内提交，再由父仓库记录子模块指针和本文档。
